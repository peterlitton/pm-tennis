# v0.8.4 release notes — Trading Performance page: header refresh + Recent Trades

**Release date:** 2026-05-06
**Type:** Trading-page UI changes (no schema changes, no worker logic)
**Scope:** 4 files modified, 1 new doc, 1 release notes
**Rollback:** Revert the touched files; no data, schema, or worker impact

---

## What changed

### Change 1 — Title

The trading page title is now "Trading Performance" (was "Trading"). Matches the operator's framing and the link on the live dashboard.

### Change 2 — Header right cluster

Two new elements added to the header, right of the date selector:

- **Live Tennis link** — quiet text link styled to match the "Performance" link added to the live dashboard in v0.8.3 (tertiary text color, dotted underline, lifts on hover). Opens `/` in a new tab.
- **Refresh button** — small grey jelly-bean pill with a circular-arrow SVG icon. Clicking does `window.location.reload()`. Replaces the prior "Refresh page for current data" footer copy.

### Change 3 — Footer collapse

Single line: `As of HH:MM AM/PM TZ · Trading v0.8.4`

- The separate "As of" line and the `Trading v0.5.1 · Refresh page for current data` row are merged into one centered line above a single horizontal rule.
- Time format aligned to operator standard: `HH:MM AM/PM TZ` (no seconds, no leading zero on hour). Was `HH:MM:SS AM/PM TZ`.
- Version number bumped to v0.8.4 (alignment with current dashboard versioning).
- "Refresh page for current data" copy removed entirely.

### Change 4 — Recent Trades section (PM-D6)

A new section rendered after Closed Contracts. Flat list of every executed transaction for the displayed date in operator's local TZ, sorted newest-first.

**Columns:**

| Column | Format |
|---|---|
| Player | Player backed |
| Time | `HH:MM AM/PM` in operator's tz |
| State | `Bought` / `Sold` (blank for settlement rows) |
| Condition | `Market` / `Limit` / `Won` / `Lost` |
| Rate | `N¢` for fill rows; `$1.00` / `$0.00` for settlements |
| Qty | Integer |
| Amount | `+$N.NN` green / `−$N.NN` red |

**Section behavior:**

- One row per executed transaction. No consolidation by player, contract, or match.
- Resting limit orders that have NOT filled do NOT appear (only printed/filled trades + settlements).
- Empty days suppress the section (matches Closed Contracts pattern).
- Date scope follows the page's date selector — viewing "Yesterday" shows yesterday's trades.

**Order-type data gap (v1 limitation):**

The Polymarket activity API does not currently expose order type (Market vs Limit) in the data we parse. v1 ships with `—` for the Condition column on fill rows (Won/Lost still populate correctly on settlement rows). Resolution requires either (a) parsing a Polymarket orders endpoint that exposes order type, or (b) cross-referencing the open-orders log to label fills. Operator-locked decision: ship v1 with `—`; revisit Market/Limit wiring in a follow-up.

## Why bundle these together

All four changes are in the same surface (the `/trading` page) and address related operator-driven needs: the section, the affordances, and the housekeeping (title, footer) all moved together to avoid touching the same files multiple times. Each can be reverted independently — see Rollback below.

## Acceptance criteria

1. Page title reads "Trading Performance".
2. Header right cluster: Live Tennis link + refresh button, in that order.
3. Live Tennis link opens https://pm-dashboard-o71w.onrender.com/ in a new tab.
4. Refresh button reloads the page on click.
5. Footer is one line: `As of HH:MM AM/PM TZ · Trading v0.8.4`.
6. As-of format matches `%-I:%M %p %Z` (e.g., `1:33 PM CDT`, not `01:33:10 PM CDT`).
7. Recent trades section renders below Closed contracts (when transactions exist).
8. Recent trades shows one row per executed transaction.
9. Sort: newest at top.
10. Resting limit orders that have not filled do NOT appear.
11. State shows `Bought` / `Sold` on fill rows; blank on settlement rows.
12. Condition shows `Won` (green) / `Lost` (red) on settlement rows; `—` on fill rows.
13. Rate shows `N¢` for fill rows; `$1.00` / `$0.00` on settlement rows.
14. Qty is integer.
15. Amount: signed dollars, green on positive, red on negative.
16. Empty days suppress the Recent trades section.
17. Mobile (≤640px): all columns visible at smaller font.
18. No regression on existing Closed Contracts table, P&L card, stats row, or date selector.

## Smoke test

After deploy, visit https://pm-dashboard-o71w.onrender.com/trading?tz=America/Chicago&token=… and check:

- Title in the header reads "Trading Performance".
- Live Tennis link visible in header right; click → new tab to live dashboard.
- Refresh button visible right of the Live Tennis link; click → page reloads.
- Footer: single line, `As of HH:MM AM/PM CDT · Trading v0.8.4`. No "Refresh page for current data" copy anywhere.
- Recent trades section renders (assuming today has at least one filled trade).
- Section rows match operator's actual day in Polymarket: same player names, same times, same buy/sell direction, same prices and amounts.
- Settlement rows (if any contracts settled today) show `Won` (green) or `Lost` (red) in the Condition column.

If the day has zero filled trades, the section should be absent (not an empty table).

## Rollback

Independent rollback per change:

- **Title only:** revert `.t-title` text.
- **Header right cluster:** remove `.t-header-right` block from template; CSS rules can stay defined.
- **Recent trades:** remove the `{% if recent_trades %}…{% endif %}` block from template AND the `recent_trades` derivation from `main.py`. CSS rules harmless.
- **Footer collapse:** restore `.t-asof` div + `.t-footer` block in the template; revert `as_of` format string in `main.py`.

Whole-bundle rollback: redeploy the v0.8.3 tarball.

## Open questions / follow-ups

1. **Order-type wiring (Market vs Limit).** The Condition column shows `—` for fill rows in v1. Closing the gap requires either an alternate Polymarket endpoint or cross-referencing the open-orders log. Decision deferred.
2. **Long-day pagination.** Days with 50+ rows make the section tall. v1 renders all rows; if operator surfaces the issue, paginate at 30 with a "show all" toggle.
3. **Conditional-fill alerting.** Operator's underlying need (knowing when a resting limit order fills while away from the dashboard) is partially addressed by Recent Trades — they can now scan for fills they didn't watch happen. A more direct alert (push notification, banner, etc.) is a follow-up not in this ship.
