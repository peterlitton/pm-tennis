# v0.8.5 release notes — Polymarket-discovered match start_time fix

**Release date:** 2026-05-06
**Type:** Bug fix (worker pipeline)
**Scope:** 2 source files modified, 1 new doc, 1 release notes
**Rollback:** Revert the two source files; no schema, API, or frontend impact

---

## What changed

Fixed a data-loss bug in the Polymarket-Global worker pipeline that caused 87% of matches on the live dashboard to display with `start_time=None`, breaking the v0.8.3 cross-status sort.

### Before this fix

Empirical state of production at 2026-05-06:

```
Total matches:   68
start_time set:   9
start_time None: 59
```

Live matches (9 of them) sorted to the bottom of the dashboard. A wall of upcoming matches with null start_times collapsed to the top, sorted only by dict-insertion-order tiebreaker. The operator's vertical-position-memory workflow — the entire reason v0.8.3 was specified — was broken.

### Root cause

Two-source dashboard. The Polymarket-only "shadow" path (matches Polymarket lists that API-Tennis hasn't matched yet) was hardcoding `start_time=None` at construction with the comment "Polymarket doesn't expose match-start time per-event in our discovery."

The comment was wrong. Polymarket Global's `startDate` field is a full ISO 8601 timestamp (e.g., `2026-05-07T06:00:00Z`). But `polymarket_global_worker.py` line 376 was truncating it at ingestion:

```python
event_date = (ev.get("startDate") or "")[:10]
```

The slice `[:10]` discarded the time portion. By the time the data reached the `pm:` shadow Match constructor, there was no time left to use, so it set `start_time=None`.

### After this fix

The full ISO timestamp is preserved at ingestion as a new `start_date_iso` field, propagated through the `slug_metadata` pipeline, and used by the `pm:` shadow Match constructor. Existing shadow matches refresh in-place on the next worker iteration (no restart required for population).

## Files changed

| File | Change |
|---|---|
| `src/polymarket_global_worker.py` | (a) Line ~376: `_fetch_us_live_matches` now preserves the full `startDate` as `start_date_iso` instead of truncating to date-only. `event_date` retained for backward compat. (b) Three slug_metadata sites (~lines 700, 718, 730) pass `start_date_iso` through. (c) Diagnostic record output (~line 1430) includes it for log consistency. |
| `src/polymarket_worker.py` | (a) The `pm:` shadow Match constructor (~line 1872) reads `meta.get("start_date_iso")` instead of hardcoding `None`. (b) The in-place update path (~line 1912) refreshes `existing.start_time` so existing shadows upgrade without a worker restart. |
| `docs/PM-D7-Start-Time-Bug-Fix-2026-05-06.md` | New file — full spec with current condition, business need, deliverable, and build instructions. |

No schema change. No API contract change. No template, CSS, or frontend change. The new `start_date_iso` field is internal to the worker pipeline; the public `/api/matches` JSON shape is unchanged (only the existing `start_time` field on each match goes from `None` to populated).

## Acceptance criteria

1. `/api/matches` returns >95% of matches with non-null `start_time`.
2. Top of the live dashboard renders the soonest-scheduled match (live or upcoming).
3. Live matches sit chronologically among upcoming matches, not segregated at the bottom.
4. `pm:`-keyed matches get a non-`None` `start_time` populated from Polymarket's `startDate`.
5. Match vertical position holds during a status transition (v0.8.3 stability property preserved).
6. No regression on API-Tennis-keyed matches.
7. Public `/api/matches` JSON shape unchanged.

## Smoke test

After deploy and worker restart:

1. ```bash
   curl -s 'https://pm-dashboard-o71w.onrender.com/api/matches' | \
     python3 -c "import json,sys; ms=json.load(sys.stdin)['matches']; \
       print(f'set={sum(1 for m in ms if m[\"start_time\"])} of {len(ms)}')"
   ```
   Expect close to 100% set.

2. Visit https://pm-dashboard-o71w.onrender.com/. The topmost match has the earliest `start_time`; the bottommost has the latest. Live matches should appear in their correct chronological slot, not clumped at the bottom.

3. Watch (or pick) an upcoming match transitioning to live in the next 5–15 minutes. Confirm it holds its vertical position across the transition.

## Rollback

Revert the two source files:
- `src/polymarket_global_worker.py`
- `src/polymarket_worker.py`

Matches with `pm:` event keys revert to `start_time=None` and sort to the top of the dashboard. The v0.8.3 sort logic continues to work; it just gets bad data again.

The doc additions can stay or be reverted independently — they don't affect runtime.

If the rollback is partial (only one of the two files), the system stays correct: change 1+2 in isolation just adds an unused field to metadata; change 3 in isolation reads a missing key from `meta` and falls back to `None` gracefully.
